# karta 1.18.0 — release test-drive checklist

A focused, incremental checklist. 1.18.0 adds one thing: **cross-item term consistency** — a binder can declare a string several work items must render identically, and karta halts a delivery whose assembled result lets one drift. The [1.17.0 checklist](v1.17.0-checklist.md) still certifies binder end-of-life, [1.16.0](v1.16.0-checklist.md) the plannotator review surface, and [1.15.0](v1.15.0-checklist.md) the evidence-and-dogfood layer.

## What 1.18.0 adds — and what this checklist is for

- **A binder can declare shared terms.** A new optional top-level `shared_terms` array names canonical strings (`id`, `canonical`, and the two-or-more `items` that must render it). `canonical` is a byte-identical *substring*, so an interpolated prefix like `binder '{slug}'` stays outside it. `validate_binder.py` schema-checks the field at plan time (kebab unique id, non-empty canonical, ≥2 items that all resolve; a dangling id errors, an item with empty `touches` warns). An absent or empty `shared_terms` is a no-op.
- **A deterministic check enforces it on the assembled tree.** New house-pattern script `skills/karta-plan/scripts/check_shared_terms.py` (pure stdlib, `--self-test`) asserts each declared `canonical` appears byte-identical in every listed item's touched files. A drift is `[FAIL]` naming the item and the string; an item not yet delivered is `[PENDING]` (skipped). Exact substring matching — no similarity threshold, no language assumptions.
- **Delivery halts on drift.** karta-deliver runs the check in the post-wave step (`deliver:waveloop` step 4) after the build/type-check; a `[FAIL]` halts the wave on the same footing as a failed post-wave build. karta-build's single-item hatch owns the same check before its merge — quoting the invocation identically (the invariant this release is about, applied to itself).
- **Planning surfaces candidates.** karta-plan's synthesis gains a `plan:terms` step: it proposes `shared_terms` from contracts that say "quote X verbatim" and from items that emit the same user-facing string, for the human to confirm. This is the only point with cross-item visibility; it is best-effort, not a guarantee. The check catches only *declared* terms — an inherent limit of a deterministic mechanism, documented, not papered over.

## Test-drive (run 2026-07-09)

Run against the shipped 1.18.0 code on `main`. Sections 1 and 2 were exercised live against `validate_binder.py` and `check_shared_terms.py` with fixture binders and trees (16/16 automated assertions passed). Section 3 was verified end to end: a fixture binder declaring a `shared_term` was delivered through a real wave in a throwaway repo — the drift case halted the wave at the post-wave check (`SHARED TERMS: DRIFT`, exit 1, revert-the-wave rewound integration to `wave-1-base`), and the corrected case completed cleanly (`SHARED TERMS: OK`, exit 0, wave tagged).

## 1. Declare and validate shared terms

- [x] Add a `shared_terms` entry to a binder and run `validate_binder.py`; confirm a well-formed entry passes, and that a dangling item id, a duplicate entry id, an empty canonical, and a single-item entry each error.
- [x] List an item with empty `touches` in an entry; confirm the validator warns without failing.
- [x] Confirm a binder with no `shared_terms` still validates and every existing binder is unaffected (the field is optional and backward-compatible).

## 2. Enforcement catches drift

- [x] Run `check_shared_terms.py --binder <path>` on a tree where every listed item renders the term identically; confirm `SHARED TERMS: OK` and exit 0.
- [x] Make one item's copy of the string differ by a character; confirm `[FAIL]` names the item and the canonical string, and the exit is non-zero.
- [x] Point an entry at an item whose touched files don't exist yet on the tree; confirm the entry is `[PENDING]` (skipped, not failed).

## 3. Enforcement in the pipeline

- [x] Deliver a binder that declares a shared term two items get right; confirm the post-wave check runs and the wave completes.
- [x] Deliver a binder that declares a shared term two items get wrong; confirm the wave halts on the drift, with the same revert-or-fix path as a failed post-wave build.
- [x] Build a one-item binder through the single-item hatch; confirm the hatch runs the same check with the invocation worded identically to karta-deliver's.
