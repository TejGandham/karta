# karta 1.19.0 — release test-drive checklist

A focused, incremental checklist. 1.19.0 adds one thing: **writer confinement** — the kaizen writer's two-place surface is now enforced by the harness, not stated in its prompt alone — plus a fix that hook's own delivery surfaced: the shared-terms check now judges an item *landed* by its git done ref, not by a touched file existing on disk. The [1.18.0 checklist](v1.18.0-checklist.md) still certifies cross-item term consistency, [1.17.0](v1.17.0-checklist.md) binder end-of-life, and [1.16.0](v1.16.0-checklist.md) the plannotator review surface.

## What 1.19.0 adds — and what this checklist is for

- **A fifth plugin hook, the fourth guard, the second fail-closed.** `hooks/scripts/guard_writer_confinement.py` runs on every `Write`/`Edit`/`NotebookEdit`. When the PreToolUse payload's top-level `agent_type` is kaizen (`karta-kaizen` bare or any `*:karta-kaizen` namespaced form — exact match, so a namesake like `karta-kaizen-v2` passes), the write target must sit under a `.karta/sme/` segment or be exactly `.karta/kaizen.json`; anything else is blocked with kaizen's own rule restated. Paths are normalized before matching (`..` traversal collapses; separators are made `/`-stable), the `.karta/kaizen.json` match is end-anchored (`.bak` variants deny), and a recognized kaizen call the guard cannot verify (no extractable string path) is denied, not waved through. Main-thread calls carry no `agent_type` and pass untouched; unknown agents pass; internal errors fail closed only on the recognized writer. Design: [`docs/specs/2026-07-09-writer-confinement-hook-design.md`](../specs/2026-07-09-writer-confinement-hook-design.md).
- **The shared-terms check implements its documented contract.** `check_shared_terms.py` used to treat "a touched file exists" as "the item landed", so an item that *edits* an existing file looked delivered before it was built — a mid-delivery run could halt a wave over wording not yet written (this actually happened in the writer-confinement delivery's first wave). An item now counts as landed only when `refs/karta/<slug>/item-<id>/done` resolves; otherwise its entries print `[PENDING] … not yet landed — no done ref` and are skipped, never failed. A non-git root makes every item pending — visible, never a crash, never a silent pass. Landed items keep byte-identity enforcement, and a landed item with no existing touched files is a FAIL.
- **The single-item hatch checks after the merge lands.** Under done-ref semantics a pre-merge check could not see the item being merged (its own ref cannot exist yet), so karta-build's 9c-single sequence now runs the check after the merge and done-ref write, halting by reverting the landed merge on a FAIL. The invocation string stays byte-identical to karta-deliver's post-wave step.
- **Asymmetry, stated:** on Codex, kaizen's confinement stays doctrine plus the OS sandbox (Codex cannot register plugin subagents; its hooks surface is not yet stable). The guard is runtime-agnostic — stdin JSON in, exit code out — so a future Codex hooks surface needs no rewrite.
