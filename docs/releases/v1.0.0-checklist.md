# karta 1.0.0 — release test-drive checklist

The path from "internally validated" to "I'd run it on a real project." Work top to bottom; each unchecked box is something you confirm yourself. The pre-flight boxes are already done for 1.0.0.

## What 1.0.0 certifies — and what this checklist is for

1.0.0 means the design and logic are stable and hardened: the pipeline, the gates and their verdicts (CONFORMANT / DEVIATION / BLOCKED / SPEC-SUSPECT), the defer/accept hatch, and git-native resume are all validated on a testbed with adversarial review and green validators.

It does **not** yet claim production mileage. This checklist closes that gap — most of all the one thing never exercised before release: a clean install from the published marketplace, run against a real repo.

## Pre-flight (done for 1.0.0)

- [x] Version is `1.0.0` in all three manifests (`.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`, `.codex-plugin/plugin.json`).
- [x] Four validators pass: `PLUGIN INTEGRITY`, `SHARED COPIES IN SYNC`, `validate_binder` (13/13), `scan_secrets` (17/17).
- [x] The marketplace manifest lists all six skills; `validate_plugin.py` guards that the manifest and the skill dirs stay in sync.
- [x] `v1.0.0` tagged and pushed.

## 1. Clean install and load — the gate never run before release

- [ ] In a throwaway environment, add the marketplace and install:
  - `/plugin marketplace add https://github.com/TejGandham/karta.git`
  - `/plugin install karta@karta`
- [ ] In `/plugin`, confirm all six skills appear: `karta:karta-plan`, `karta:karta-deliver`, `karta:karta-build`, `karta:karta-verify`, `karta:karta-validate`, `karta:karta-plainlanguage`.
- [ ] Confirm both gate agents are present: `karta-acceptance-reviewer`, `karta-safety-auditor`.
- [ ] Invoke `karta-plan` from the **installed** plugin (not the dev repo) and confirm it loads and runs.

## 2. Smoke test the pipeline end to end

- [ ] On a small real repo, run `karta-plan` on a 3–5 item feature. Confirm it produces a validated binder and commits only when you say "commit".
- [ ] Run `karta-deliver`. Confirm it builds in parallel waves onto `karta/<slug>/integration`, gates each item, and ends at the integration branch — no PR, no push.
- [ ] Review the integration branch and confirm the work is correct and mergeable.
- [ ] Confirm the run report reads plainly — the plain-language standard in action.

## 3. Exercise the recovery and gate paths

- [ ] **Kickback:** give one item an oracle it fails. Confirm DEVIATION kicks back to build, and the 2-attempt cap halts with a call to action.
- [ ] **Halt hatch:** at a halt, confirm the Phase-4 choice (fix-and-rerun / accept / defer / revert the wave) is offered, and that accept and defer behave as documented (accept merges with the waiver trailers; defer leaves the item not-done and hands off incomplete).
- [ ] **Resume:** kill a run mid-wave. Re-run `karta-deliver`, confirm it detects leftovers and offers resume/clear, and that resume skips items already done.
- [ ] **Empty diff:** point an item at work already present (or that produces nothing). Confirm it halts BLOCKED-empty rather than passing.
- [ ] **Safety gate:** make an item cross a boundary it never declared. Confirm the safety auditor flags it and escalates to you after its cap.

## 4. Breadth and scale — the claims least proven

- [ ] Run at least one **non-frontend** stack — a backend endpoint, a CLI command, or a data change — to test the stack-agnostic claim beyond the testbed and the frontend path.
- [ ] Run a wider binder (8+ items, some with dependencies) to watch the serial merge queue and parallel waves under load on a real repo.
- [ ] On a UI item, confirm the visual gate (`karta-validate`) brings up the app and compares against the design.

## 5. Cleanup

- [ ] After a run, confirm worktrees, the `karta/<slug>/*` branches and tags, and the `refs/karta/<slug>/*` refs either clean up or are clearly listed so `clear` can remove them.
- [ ] Confirm no `.karta/` state or stray worktree is left somewhere unexpected.

## If something fails

karta surfaces every halt with a call to action and never opens a PR or pushes on its own. A failed run leaves the integration branch and the item worktrees in place for inspection — read the halt report, fix or re-plan, and re-run (resume is git-native). File anything surprising against the repo before trusting karta with bigger work.
