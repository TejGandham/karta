# karta 1.16.0 — release test-drive checklist

A focused, incremental checklist. 1.16.0 adds one thing: **plannotator as an optional review surface** in karta-plan's binder review and karta-deliver's branch review, ported from keel's proven pattern. Nothing else changed; the [1.15.0 checklist](v1.15.0-checklist.md) still certifies the evidence-and-dogfood layer and [1.14.0](v1.14.0-checklist.md) the enforcement chain.

## What 1.16.0 adds — and what this checklist is for

- **A second review surface, gated on a deterministic probe.** Both skills probe for the separately-installed plannotator CLI with one cross-shell line (`uv run python -c "import shutil,sys; sys.exit(0 if shutil.which('plannotator') else 1)"`). A failed probe is silent: plannotator is never mentioned, and the flows run exactly as 1.15.0 shipped them.
- **karta-plan**: on a successful probe, a recommendation-first offer — review the drafted binder as the in-chat editable card, or as one browser annotation session over the same card fields (rendered to session scratch, never committed). Annotations map onto binder-field edits, the validator re-runs, and an annotation that maps to no field unambiguously is asked back in chat — never guessed. The surface changes no gate and no verb: commit still takes the explicit `commit`.
- **karta-deliver**: the report's review pointer gains an offer to open the integration branch's diff in a `plannotator-review` session. The branch stays the user's to merge.

- [x] On a host without the plannotator CLI, run a `karta-plan` through to the review step; confirm the card flow runs with zero mention of plannotator. [Codex 1.19 evidence](../showcase/codex-1.19-compatibility/plannotator.md#plan-review-probes)
- [x] Install plannotator, repeat; confirm the recommendation-first offer appears with exactly the two verbs (`card`, `plannotator`). [Codex 1.19 evidence](../showcase/codex-1.19-compatibility/plannotator.md#plan-review-probes)

## 2. Annotation session round-trip

- [x] Choose `plannotator`; confirm the rendered scratch file carries the binder frame plus every work item's card fields (title, summary, contract, touches, oracle, depends_on, flags) in dependency order, and lives outside `.karta/binders/`. [Codex 1.19 evidence](../showcase/codex-1.19-compatibility/plannotator.md#rendered-review-file)
- [ ] Annotate one item's contract and one unmappable free-form note; confirm the contract edit lands in the binder (validator re-run passes) and the free-form note comes back as an in-chat question against that item.
- [ ] Confirm nothing commits until the explicit `commit` verb, and the scratch file is not in the commit.

The [Codex 1.19 annotation run](../showcase/codex-1.19-compatibility/plannotator.md#human-annotation-round-trip) mapped summary fields and returned an ambiguous heading note without guessing. It did not map a contract field, so the contract-specific box remains open. It also proved that no implicit commit occurred, but it did not run the explicit `commit` verb and inspect the resulting commit, so that box remains open too.
