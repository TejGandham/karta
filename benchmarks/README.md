# karta framework bench

A periodic, deterministic-first benchmark of the karta plugin itself — not of any consumer project. Source of truth: [`bench-spec.json`](bench-spec.json). The pre-existing [SME A/B bench](sme/README.md) is one family of this bench and keeps its own charter.

## 1. What this bench is

The bench measures karta version-over-version across seven axes: execution-flow enforcement gaps, SME pack appropriateness and fillable gaps, dark-area stabilization, per-phase performance, output quality, doc/repo parity, and field adoption. Every result is a committed, machine-diffable file. The doctrine matches karta's own: enforced checks and frozen denominators over skippable prose, and every metric must stay honest under growth — append-only ledgers, per-cohort attribution, epoch joins for LLM arms. It was assembled 2026-07-17 against karta 2.21.0, with `/mnt/agent-storage/vader/src/parchmark` and `/mnt/agent-storage/vader/src/gringotts` as the consumer field repos.

Chassis decision, in one paragraph. promptfoo is the suite chassis for the executable and judged arms — it is MIT-licensed and free; the research verdict "buy" means adopt, not purchase. It will be pinned via `benchmarks/bench/package.json` when the executable lane lands, invoked as `npx promptfoo@<pin>` with `PROMPTFOO_DISABLE_TELEMETRY=1`. The system under test is driven through promptfoo's exec provider wrapping a literal headless `claude -p` with a bench-controlled `CLAUDE_CONFIG_DIR` — not the claude-agent-sdk provider — because karta is a plugin whose value is skills plus six hook guards firing under real settings, and only a real Claude Code process exercises that; it rides the owner's subscription auth and needs no Docker on the default path. The deterministic lane never depends on promptfoo or node: every deterministic vector is a standalone stdlib-python script, composed by one blocking gate runner (`gate/run_gate.py`). Vendored `bayes_evals` supplies paired credible intervals only at k>=5; every n<=3 arm reports raw counts. Two small owned scripts (`miner.py`, `trend.py`) fill the two gaps no surveyed tool provides: per-phase telemetry and git-native longitudinal trending. Rejected chassis: Harbor and Inspect AI (Docker-sandbox-first fights this host's Incus-only Docker), crucible (anonymous alpha, dead), skill-creator evals (subagent-scoped; machinery cribbed as components).

## 2. Layout

Everything lives under `benchmarks/` in the karta repo. Per-family directories hold one vector card per vector, `<family>/<id>.md`.

- `flow/` — execution-flow vectors (2 cards) plus the committed seed audit map [`flow/seed-map-2026-07-17.json`](flow/seed-map-2026-07-17.json)
- `dark/` — dark-area vectors (3 cards)
- `field/` — consumer-repo field vectors (3 cards)
- `sme/` — the pre-existing SME A/B bench, embraced as-is: [`sme/README.md`](sme/README.md), `sme/run.py`, `sme/fixtures/`, `sme/results/`, plus 2 vector cards
- `perf/` — performance vectors (3 cards)
- `quality/` — output-quality vectors (3 cards)
- `parity/` — doc/repo and mirror parity vectors (2 cards)
- `sec/` — untrusted-input vector (1 card)
- `meta/` — bench-about-the-bench vectors (5 cards)
- `fixtures/` — shared fixtures with one sha-pinned authority, [`fixtures/REGISTRY.json`](fixtures/REGISTRY.json); first entry is the frozen [gringotts browse-refinements binder](fixtures/gringotts-browse-refinements-binder-2026-07-17.json)
- `epochs/` — [`epochs/epochs.json`](epochs/epochs.json), the model-epoch authority every LLM-arm comparison joins against
- `gate/` — the deterministic gate lane: `run_gate.py`, the blocking pre-tag entrypoint
- `probes/` — deterministic probe scripts, one `probes/<id>.py` per vector (OpenSSF-Scorecard probe style; the card at `<family>/<id>.md` is its authority), populated as vectors land
- `results/` — committed outputs: `results/gate/` for per-version gate verdicts, `results/<YYYY-MM-DD>-v<semver>/` for reduced bench-cycle runs, `results/TREND.md` regenerated by `trend.py`
- `research/` — buy-vs-build research record [`research/evaltool-2026-07-17.json`](research/evaltool-2026-07-17.json)
- `bench/` — the chassis toolchain (package.json pinning promptfoo, `claude_arm.sh`, `miner.py`, `reduce.py`, `trend.py`, vendored `bayes_evals.py`), built when the executable lane lands

## 5. How to run today

```
python3 benchmarks/gate/run_gate.py
```

This is the deterministic lane: zero model calls, no node, no promptfoo. It composes whatever probe scripts exist; every vector not yet built reports a loud SKIPPED row — never a silent green. Gate verdicts land as dated JSON under `benchmarks/results/gate/` (one file per version; the precommit gate refuses a version-bump commit unless the current version's file exists, is green, and matches the RC sha). Vector-local evidence lands in each family's own results directory. Full bench-cycle runs, when the chassis lane exists, reduce to `benchmarks/results/<YYYY-MM-DD>-v<semver>/result.json` + `summary.md`, and `trend.py` regenerates `benchmarks/results/TREND.md` from every committed result file.

## 6. Build-out status

The spec sequences the first cycle as build-out, not measurement, in three phases after a blocking seeding step:

- Phase 0 — seeding (blocking): commit the session artifacts before the scratchpad expires — the audit map to `flow/seed-map-2026-07-17.json`, the tool research to `research/evaltool-2026-07-17.json`, the frozen gringotts fixture, `fixtures/REGISTRY.json`, `epochs/epochs.json`.
- Phase 1 — gate lane: `gate/run_gate.py` + the precommit version-bump block, composing the deterministic vectors as they land, with the fixture registry and epochs file as shared authorities.
- Phase 2 — field lane: the consumer-repo audits, the adoption ledger (manifest seeded back to 1.8.0), companion-loop health, telemetry modes 1+4, and ownership-cost-ledger row 1 (which resolves the promptfoo/bayes contradiction as its first buy-vs-build entry). Also: set `cleanupPeriodDays>=365` — transcript decay is already destroying evidence.
- Phase 3 — fixture/quarterly arms, gated on a one-time pilot of the headless `claude -p` deliver path: midwave-crash, sme-fixture-campaign, fixture-cost-baseline (whose first run IS the 2.21.0-era baseline), verify-drills Families E/K, doc-gardner-recall, sec P3, bench-self-integrity M1. First-run outputs bootstrap all absolute budgets, freeze all fixture shas, pin oracle shas, and record the initial alias-to-model map.

Today: phase 0 is seeded (spec, seed map, research record, fixture registry, epochs file, and all 24 vector cards committed) plus 2 thin probes. The gate lane is next; expect mostly SKIPPED rows from `run_gate.py` until phase 1 lands.

## 7. Honesty rules

All 15, verbatim from the spec.

1. INHERITED VERBATIM from benchmarks/sme/README.md and applied bench-wide to every A/B and cost claim: 'Report only the A/B delta on these fixtures. Never print a per-repo "you saved X lines/tokens here" number: in a live repo the unbuilt version was never written, so there is no real baseline to subtract from. The only honest per-repo figure is the karta-debt ledger (a counted list of real markers).'
2. Every result file records judge model ID, temperature, rubric versions, and prompt hashes; any change to judge or rubric is a declared METRIC RESET in TREND.md — cross-reset comparisons are never printed.
3. Deterministic floor before judge: a trial failing any automated assert scores 0 on that task regardless of judge opinion; the judge never overrides or rescues an automated fail.
4. The frozen calibration set is re-scored every chassis run; if calibration scores move beyond tolerance, the run is flagged RULER-CHANGE and subject deltas from that cycle are not reported as regressions or wins.
5. No single-run numbers, ever: every reported metric carries k (or n/N with the underlying list) and spread; deltas below the ~3pp agentic-noise floor are labeled NOISE, deltas without interval support are labeled NO-DETECT — both appear in the committed file, not silently dropped. At n<=3, raw count tables only — no credible intervals.
6. A task is admitted only after its oracle solution passes its own verifier; a task whose oracle stops passing is QUARANTINED and excluded from aggregates until fixed, with the quarantine noted in the results.
7. Every executable trial starts from a fresh git init of the fixture template; a trial found to have read prior-trial state or real repo history is VOIDED and rerun (acutely relevant to git-native karta).
8. The evidence layer (per-trial verdicts, probe findings) is always committed alongside the score layer; aggregates and weights can be recomputed later, evidence cannot be reconstructed — scores may be re-derived but never replace their evidence.
9. Version-over-version claims come only from same-task, same-rubric-version, same-epoch pairs; tasks added later enter the trend at their added_in version and are never backfilled or imputed.
10. Safety axis preserved from the sme precedent: a smaller diff that fails its acceptance gate LOSES — no vector may trade correctness for a better-looking number, and static-judged acceptance is always labeled weaker evidence than an executed gate.
11. SHARED COMPARABILITY CONVENTION (extracted once so 22 vectors stop re-deriving it; every vector card references these): (a) denominators are frozen at first commit or era/cohort-partitioned — a metric whose denominator grows with the manifest/corpus/history is reported as per-item status transitions or new-cohort-only trends, never as a raw count delta; (b) ledgers, corpora, claims files, and probe registries are append-only, with retirements visible in the diff; (c) release-over-release comparisons run on the intersection of stable IDs with the previous committed run; (d) LLM-arm deltas require an epoch join against benchmarks/epochs/epochs.json; (e) fractions are printed as n/N with the underlying item list, never bare percentages at small N.
12. Never-exercised flows report 'n/a (fixture-only)', never '0 violations' — a zero over an empty exercise history is not a pass (field baseline: failed/accepted/kickback at 0 occurrences across 54 items).
13. Anchors into moving prose are content-strings or sentence-regexes, never line numbers; an anchor miss is a loud ANCHOR-LOST/ANCHOR-STALE state requiring same-commit triage, never a silent pass.
14. Cross-project and cross-corpus deltas print only with an 'advisory — confounded' label; honest version-over-version cost claims come from the same project or the frozen fixture.
15. Fixture-contamination honesty: absolute scores on public-pattern fixtures are not capability claims; only same-fixture version-over-version deltas are.

## 8. Comparability convention

The shared discipline every vector card references instead of re-deriving (rule 11 above, spelled out):

- Frozen denominators: a denominator is frozen at first commit or partitioned by era/cohort. A metric whose denominator grows with the manifest, corpus, or history is reported as per-item status transitions or new-cohort-only trends — never as a raw count delta.
- Append-only ledgers: findings ledgers, corpora, claims files, and probe registries only grow; retirements stay visible in the diff, with resolutions carrying the resolving commit hash.
- New-cohort-only trending: items added later enter the trend at their added_in version and are never backfilled or imputed.
- Intersection of probe ids: release-over-release comparisons run on the intersection of stable probe/finding/case IDs with the previous committed run.
- Epoch joins for LLM arms: any delta involving a model-driven arm joins on the epoch id in `epochs/epochs.json`; results from an alias-remap cycle are quarantined until the new epoch baseline is committed.
- Honest fractions: n/N with the underlying item list, never bare percentages at small N.

## 9. Open questions

All 10 from the spec, compressed; full text in `bench-spec.json`.

1. Chassis contradiction: research says buy promptfoo + vendor bayes_evals, sharpened vectors dropped both — the reconciliation needs an explicit accept/reject row in ownership-cost-ledger or the next author reintroduces them ad hoc.
2. Fixture registry ownership: `fixtures/REGISTRY.json` needs one owner and a phase-1 build slot, or the shared fixtures fork silently and bench-self-integrity M2 has no authority to audit against.
3. Headless `claude -p` karta-deliver is unproven: one pilot gates four vectors — if it fails, which accept the manual-capture fallback and which are deferred?
4. Per-release wall-clock budget: 20 of 22 vectors have an every-release component and the <10 min gate-lane budget is unvalidated — which lanes demote to quarterly if it blows?
5. platform-native.md ruling: confirm and commit the exceptions.json skip so validate_packs stops flagging it every run.
6. serve_status `--snapshot-json`: add proactively (small diff, de-risks two vectors, natural home for the sec P1 escaping fixes) or only on first breakage of the pinned import probe?
7. Unmatched-ecosystem SME gaps: when does an unmatched ecosystem (JVM/Swift/ML) warrant a new built-in pack — human judgment over the list, or a threshold rule?
8. Kaizen phase 2: field-companion-loop-health is the signal collector — define the threshold trend that triggers build vs retire.
9. Attempt-cap persistence: verify-drills sub-probe (c) is expected to fail structurally (caps live in orchestrator memory) — fix via a ref-carried counter before or after the first quarterly run documents it?
10. Transcript retention: `cleanupPeriodDays>=365` plus mining within 7 days is a host precondition outside the repo — enforce it as an ownership-cost-ledger manual-step row or a gate-lane environment check?
